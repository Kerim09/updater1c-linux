# updater1c-linux-stage3

Третий прототип Linux-аналога Обновлятор-1С.

Исправления:
- персональный логин/пароль у каждой базы;
- при переключении базы логин/пароль подставляются из выбранной базы;
- принудительно добавлены Copy/Cut/Paste/Select All в поля ввода;
- корректный импорт web-баз из ibases.v8i (`WS=...`), без превращения в `/FWS=...`;
- web-базы запускаются через `1cestart ... /WS<url>` или через браузер;
- проверка настроек пытается определить имя и версию конфигурации через `DESIGNER /DumpConfigToFiles`;
- добавлена кнопка `Скачать обновления` через `update-api.1c.ru`;
- скачивание учитывает последовательность обновлений `upgradeSequence`, которую возвращает сервис обновлений 1С;
- скачанные zip-архивы распаковываются в папку обновлений.

## Запуск

```bash
sudo apt update
sudo apt install python3 python3-venv unzip postgresql-client -y
unzip updater1c-linux-stage3.zip
cd updater1c-linux-stage3
./run.sh
```

Если PySide6/Qt ругается на XCB:

```bash
sudo apt install libxcb-cursor0 libxcb-xinerama0 libxkbcommon-x11-0 -y
```

## Важно про обновления

Для скачивания обновлений в свойствах базы желательно заполнить поле:

```text
Код программы обновлений 1С
```

Например для Бухгалтерии обычно используется `Accounting`.
Если поле пустое, программа попробует подобрать код по имени конфигурации.

Логин/пароль ИТС задаются на вкладке `Настройки программы`.

## Важно про определение версии

Определение конфигурации/версии работает только для файловых и клиент-серверных баз, доступных через Конфигуратор.
Для web/1cfresh-баз Конфигуратор по URL недоступен — там версию нужно заполнять вручную или получать отдельным механизмом.

## Исправления fix1

- Поля ввода теперь используют собственный `ClipboardLineEdit`.
- `Ctrl+V/C/X/A` обрабатываются напрямую.
- В контекстном меню вставка всегда доступна; если Qt не видит буфер, используется fallback через `wl-paste`, `xclip` или `xsel`, если они установлены.
- Определение конфигурации переписано: теперь читается только корневой `Configuration.xml` и только блок `Configuration/Properties`, чтобы не подхватывать случайные значения вроде `ru`.
- Поле `Код обновлений` больше не заполняется русским наименованием конфигурации. Если код не распознан по известной мапе, его нужно указать вручную.
- Невалидный старый код обновлений из предыдущей версии очищается при загрузке конфига.

Для лучшей работы вставки в Wayland/X11 можно поставить:

```bash
sudo apt install wl-clipboard xclip xsel -y
```

## Исправления fix2

- Проверка конфигурации больше не делает fallback на иерархическую выгрузку.
- Используется только `/DumpConfigToFiles <dir> -Format Plain`.
- Временная папка сделана короткой: `/tmp/u1c/cXXXXXX`.
- Добавлены `/DisableStartupDialogs` и `/DisableStartupMessages`, чтобы 1С не открывала модальное окно с ошибкой и писала диагностику в лог.
- Если плоская выгрузка не дала `Configuration.xml`, программа не зависает и не пытается выгружать всю конфигурацию длинными иерархическими путями.
## Stage 4 Metadata

Добавлены два метода определения конфигурации/версии:

1. `execute` — через внешнюю обработку `.epf`, которая внутри базы читает `Метаданные.Имя`, `Метаданные.Синоним`, `Метаданные.Версия()`.
2. `dump` — через `DumpConfigToFiles -Format Plain`.
3. `auto` — сначала пробует `.epf`, затем fallback на `DumpConfigToFiles`.
4. `manual` — ничего не проверяет, используются поля в свойствах базы.

Шаблон кода для обработки лежит в `tools/metadata_probe`.

## DbInfo fix1

Исправлена передача параметра результата в обработку:

```bash
/Execute DbInfo.epf /C U1C_META_OUT=/tmp/u1c_meta_xxx.json
```

В `tools/metadata_probe/ManagedFormModule_no_sync_client.bsl` лежит исправленный модуль формы обработки без вызова `КаталогВременныхФайлов()` на клиенте. В управляемом клиенте синхронный вызов `КаталогВременныхФайлов()` может быть запрещен, поэтому обработка теперь требует путь результата через `/C`.

## DbInfo fix2

Исправлена схема передачи параметра в 1С:

```bash
/Execute DbInfo.epf /CU1C_META_OUT=/tmp/u1c_meta_xxx.json
```

То есть `/C` и значение передаются одним аргументом.  
Дополнительно приложение передает переменную среды `U1C_META_OUT`.

Новый модуль формы обработки:

```text
tools/metadata_probe/ManagedFormModule_fix2.bsl
```

Если открыть обработку вручную без `/C`, она не должна закрывать 1С. Она просто покажет имя, синоним и версию через `Сообщить()`.

## Stage 7 Update API

Доработан блок скачивания обновлений по материалам `v8platform/updateApiClient`:

- `update/info` теперь вызывается в правильном формате: `programVersion: {programName, versionNumber}`, `updateType`, `platformVersion`;
- для обычных обновлений используется `NewConfigurationAndOrPlatform`, чтобы не перескакивать на другую редакцию без отдельного разрешения;
- загрузка файлов обновлений выполняется через `configurationUpdateDataList`, с учетом `templatePath`;
- zip распаковываются с учетом CP866-имен старых 1С-архивов;
- добавлена проверка `hashSum`, если API вернул MD5/SHA256;
- добавлена операция `Ещё... → Обновление → Скачать новую платформу 1С`;
- скачивание платформы использует `platformUpdateResponse.distributionUin`, затем `platformDistributionUrl` из `/update-platform/programs/update/`.

## Stage 8 Auto Update

Добавлено:

- плоская распаковка обновлений: `<updates_dir>/<Program>/<release>/1cv8.cfu`;
- архив ZIP после успешной распаковки удаляется;
- `latest_update_info.json` хранится в папке программы обновлений и задает порядок `upgradeSequence`;
- кнопка `Обновить выбранную базу` запускает обновление по скачанным релизам в порядке API;
- цепочка на каждый релиз: `/UpdateCfg` → `/UpdateDBCfg` → запуск обработчиков обновления через ENTERPRISE `/C ЗапуститьОбновлениеИнформационнойБазы`;
- перед обновлением предлагается резервная копия;
- файловая база архивируется как `1Cv8.1CD.tar.gz`;
- серверная база может копироваться через `.dt`, PostgreSQL `pg_dump`, MSSQL `.bak` через `sqlcmd`;
- при ошибке выполняется попытка отката из созданной копии;
- для скачивания платформы добавлена форма выбора `distributionUin`, которые вернул update-api, и желаемых типов дистрибутивов.

## Stage 16 LibGCC Preload

Исправление для ошибки запуска 1С на новых Ubuntu:

```text
/opt/1cv8/.../1cv8: /opt/1cv8/.../libgcc_s.so.1: version `GCC_12.0.0' not found
/opt/1cv8/.../1cv8: /opt/1cv8/.../libgcc_s.so.1: version `GCC_13.0.0' not found
```

Причина:
платформа 1С подхватывает старую `libgcc_s.so.1` из своего каталога, а системная библиотека `libhwy.so.1` требует более новые символы GCC.

Что сделано:
- при любом запуске 1С приложение автоматически добавляет:
  ```text
  LD_PRELOAD=/lib/x86_64-linux-gnu/libgcc_s.so.1
  ```
  или другой найденный системный путь;
- это применяется к интерактивному запуску, Конфигуратору, `/Execute`, `/UpdateCfg`, `/UpdateDBCfg`, резервным копиям и другим пакетным операциям;
- файлы в `/opt/1cv8` не переименовываются и не удаляются.

Проверочная ручная команда:
```bash
LD_PRELOAD=/lib/x86_64-linux-gnu/libgcc_s.so.1 /opt/1cv8/x86_64/8.3.27.2130/1cv8 DESIGNER /F/mnt/Data/bases/DemoDocMngCorpMK
```
